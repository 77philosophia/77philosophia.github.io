主动视觉

主动视觉的概念主要由 阿洛伊蒙纳斯（John (Yiannis) Aloimonos） 及其同事在 1987年 左右明确提出。

几乎在同一时期，其他研究者也从不同角度贡献了类似的理论：

  • John Aloimonos: 在其论文 Active Vision 中首次正式确立了这一学术术语。
  • Dana Ballard: 提出了 “拟人视觉”（Animate Vision），强调视觉系统应模仿人类的眼球运动和注意力机制。
  • Ruzena Bajcsy: 提出了 “感知控制” 的思想，认为感知不应是被动的数据收集，而是一个受控的过程。

定义主动视觉

​ View Path Planing(VPP)或者Next-Best-View(NBV)

​ 主动视觉是指观测者（相机或机器人）根据任务需求，主动、有目的地改变其传感器（镜头）的参数（如位置、方向、焦距、光圈等），从而获取更优质、更易处理的视觉信息流。

核心任务——三维重建和主动感知

主动三维重建(active 3D reconstruction)

获取物体全局、完整、高精度的三维几何表征（如点云、Mesh、3DGS）。

全局信息增益（Information Gain）。机器人需要规划一条路径，把物体所有的死角、阴影、凹槽全部扫干净。

主动三维重建目前的发展感觉是不是已经sota了，主要是精度的问题。

2026的一篇工作：https://drive.google.com/file/d/1qA3WS7Z3KPYxNYwlfsfKqLNi5yddjg9Q/view?pli=1

边操作边转换视角(Active Manipulation / In-hand Visual Servoing)

目标： 服务于当前的操作任务（如抓取、装配、插拔）。

局部关键特征的可见性。算法不需要知道整个物体长什么样，它只关心操作点（ROI）有没有被遮挡、有没有对齐、有没有放大到足够清晰的程度。

[!NOTE]

我们主要要解决的是第二个任务。

两种思路：

1.局部重建为操作服务：机器人在边操作边转换视角时，往往也会在后台实时构建一个局部的、精细的神经辐射场（如 Live-3DGS）。它通过快速的视角转换，瞬间脑补出操作区域的局部 3D 结构，从而辅助计算接触力学。

2.统一的policy网络，比如pi-0之类的端到端模型已经不再区分“我现在是在看，还是在动”。

思路一：基于vla模型做微调整

任务目标：“桌面级操作”（Desktop Manipulation）。"Pick up the DB connector and show its pins to the camera"。

模型选择：

  • Openvla: 把动作离散化（Quantization）成一个个像单词一样的 Token（比如 0-256 之间的整数）。泛化能力极强，控制频率低，不够细。几hz
  • Act（丝滑速度）: 有action chunk,用 CVAE（条件变分自编码器）来建模人类演示的多样性。频率高，适合丝滑高难度动态任务。20hz~50hz,但是几乎没有泛化能力，它不认识什么是“DB接口”，如果你把桌子换个颜色，或者把接口挪远了 20 厘米，它可能就完全抓瞎，需要重新采集数据。
  • diffusion policy（多峰分布）: 它把机器人的动作轨迹看作是一张“噪声图像”。模型输入当前的视觉特征，然后通过多轮“去噪（Denoising）”。擅长处理多峰分布。和 ACT 一样，属于端到端策略网络，缺乏高层的语义理解和跨物体的通用泛化能力。此外，由于需要多步去噪迭代，在算力较弱的设备上可能会带来轻微的推理延迟。
  • 目前：VLA + Policy Head（强强联合）
    • 把 OpenVLA 当作眼睛和脑袋，把 ACT 或 Diffusion Policy 当作小脑。

主机外设通信方式和常见问题总结

Posted on 2026-05-07 In 专业笔记

一：各种外设通信方式、特点、问题和方案

在机器人开发中，将各种不同协议的外设集成到一起时，物理连接只是第一步，稳定性调试才是最耗时的。

1.网络类外设(比如vr,谷歌眼镜)

用来摇操。

这类设备最大的敌人是延迟（Latency）*和*抖动（Jitter）。

谷歌眼镜通过网络发送数据时，如果主机开启了防火墙，或者路由器分配了动态 IP（DHCP）导致 IP 变化，会导致连接中断。通常需要设置静态 IP。

2.串行/执行类外设 (如：SO-101 电机总线)

这类设备直接关系到机器人的动作安全。

端口跳变 (Port Flapping)：

  • 现象： 今天是 /dev/ttyACM0，拔插一下变成了 /dev/ttyACM1。
  • 对策： 机器人专家通常使用 udev rules，通过设备的 Vendor ID 和 Product ID 绑定一个固定的别名（例如 /dev/arm_motors），无论怎么拔插都不会变。

电源干扰 (EMI)： 电机转动时会产生巨大的电磁干扰。如果 USB 线没有屏蔽环（那个圆柱体），可能会导致串口通信瞬间断开，程序报错崩溃。

权限问题： 刚装好的系统默认普通用户没有串口读写权限，经常报错 Permission denied，需要将用户加入 dialout 组。

3.视觉类外设 (如：OpenCV 相机)

这类设备消耗最大的系统资源。

USB 带宽超限：

  • 现象： 插一个相机没问题，插两个相机其中一个就黑屏或掉帧。
  • 原因： 主机主板上的多个 USB 接口往往共享一个 USB Controller。两个 1080P/60fps 的原始流会撑爆总线带宽。

索引混乱： cv2.VideoCapture(0) 里的 0 并不总是你想要的那个相机。一旦增加新摄像头，索引可能会乱掉。

通常的解决方法：

1.使用带电源的 Hub： 如果你的机械臂上有多个相机，务必买那种自带 DC 插头的 “有源 USB 3.0 Hub”。这样电量是由插座提供的，主机的 USB 只负责传输信号。

（多相机并行： 如果你在一个 USB Hub（集线器）上插了 3 个摄像头。虽然 Hub 连在电脑上，但如果这个 Hub 没有外接电源，它必须把主控口的那点电流分给 3 个相机。这会导致每个相机分到的电流不足，出现图像闪烁、掉帧、甚至设备反复掉线的情况。）

（USB 线如果超过 2-3 米，线缆本身的电阻会导致电压降（Voltage Drop）。到达相机端的电压可能只有 \(4.5\text{V}\)，这足以让很多工业相机罢工。）

2.电源管理关闭： Linux 默认会有 USB 省电策略（Auto-suspend），这会让相机在不工作时断电。

4.HID 交互类外设 (如：SpaceMouse)

这类设备的核心问题在于驱动兼容性。

  • 内核抢占： Linux 有时会把 SpaceMouse 识别为一个普通的“三键鼠标”并试图让它控制光标。这会导致你的 Python 程序（如 pyspacemouse）无法独占设备。
  • 休眠唤醒失败： 很多 USB 外设在电脑进入省电模式后再唤醒，会导致驱动挂死，必须重新插拔。

经常需要重新插拔，程序有的时候需要加心跳重连机制。

二： usb2.0/3.0端口

在机器人开发和高精度工业场景中，选择 USB 2.0 还是 3.0 对外设的影响远不止“速度快慢”那么简单，它直接关系到数据的确定性、设备的稳定性和系统的整体功耗。

2.1 传输带宽与数据通量 (Bandwidth)

这是最直观的区别，直接决定了传感器能给出的“信息密度”。

  • USB 2.0 (\(480\text{ Mbps}\)):
    • 影响： 限制了视频流的质量。如果你在 2.0 接口上接高清相机，你可能被迫只能选择较低的分辨率（如 \(640 \times 480\)）或较低的帧率（\(15\text{ fps}\)）。
    • 典型后果： 画面出现明显的运动模糊（Motion Blur），对于需要 ±0.025mm 级精度的视觉引导（Visual Servoing）来说，低分辨率和丢帧是致命的。
  • USB 3.0 (\(5\text{ Gbps}\) 及以上):
    • 影响： 能够支持原生未压缩图像（RAW）传输。
    • 典型后果： 画面更清晰、延迟更低。深度相机（如 RealSense D435）必须在 3.0 模式下才能输出高分辨率的深度图，否则只能降低采样精度。

2.2 供电能力 (Power Supply)

如前所述，这直接影响外设能否“跑得稳”。

  • USB 2.0 (\(500\text{ mA}\)): 很多现代传感器（如高性能激光雷达、带计算单元的相机）在启动瞬间的峰值电流会超过 \(500\text{ mA}\)。
    • 影响： 设备可能频繁重启、无法识别，或者在工作负载加大时（例如相机开始高频闪光或计算）突然掉线。
  • USB 3.0 (\(900\text{ mA}\)): 提供了更充裕的功率缓冲。
    • 影响： 能够支持更多无需外接电源的小型机器人外设（如小扭矩伺服控制板）。

2.3 通信机制：半双工 vs 全双工 (Duplex)

这是一个经常被忽略的技术细节，影响控制的实时性。

  • USB 2.0 (半双工): 就像单行道，数据在同一时间只能往一个方向走。主机必须不断地去轮询（Polling）外设是否有数据。
    • 影响： 增加了微小的指令延迟（Latency）。对于需要极高实时反馈的力控算法，这种轮询机制可能引入非确定性的时间抖动。
  • USB 3.0 (全双工): 支持同时发送和接收数据。
    • 影响： 通信更高效，降低了“排队等待”的时间，非常适合需要双向高频交互的设备。

2.4 关键干扰问题：2.4G 信号杀手

这是 USB 3.0 在机器人领域的一个著名“天坑”。

  • 现象： 当你使用 USB 3.0 接口传输高带宽数据时，它会产生一种宽频带电磁干扰。
  • 影响： 这个干扰恰好覆盖了 2.4GHz 频段。
  • 后果： 如果你的主机（如 Jetson Orin）插着 USB 3.0 的相机，那么旁边的 2.4G WiFi 信号或无线手柄收发器可能会瞬间断连、信号极差或丢包严重。这解释了为什么有些谷歌眼镜或无线手柄在连接 3.0 设备时必须强制切换到 5G 频率或使用加长的屏蔽线。

2.5 线缆长度限制

USB 2.0: 相对宽容，普通线缆可以做到 5 米左右仍保持稳定。

USB 3.0: 由于信号频率极高，衰减非常快。普通的 passive（无源）线缆超过 3 米就会出现严重的信号完整性问题。

  • 影响： 如果你的机械臂较大，需要从基座拉长线到末端相机，3.0 的线必须使用有源延长线 (Active Cable) 或是昂贵的工业屏蔽线。
特性 USB 2.0 USB 3.0 对机器人的建议
电机/SpaceMouse ✅ 完全足够 ⚠️ 浪费且易受干扰 建议插在 2.0 口，预留 3.0 给视觉
工业/深度相机 ❌ 延迟高、易掉帧 ✅ 必须使用 必须使用 3.0 且注意线缆屏蔽
无线接收器 ✅ 较稳定 ❌ 严重干扰 尽量远离 USB 3.0 插槽或使用延长线隔离

在做 LeRobot 这种涉及视觉和精确动作控制的项目时，通常的配置策略是：视觉走 3.0（保证感知密度），控制走 2.0（保证链路纯净稳定）。

GPU 的工作原理：从并行像素到 CUDA 加速

Posted on 2026-05-06 In 专业笔记

1. Gpu 和 cpu

  • CPU 尤其擅长处理涉及分支逻辑、操作系统管理和顺序算法的任务。
  • GPU 是专为并行计算而设计的。它不是由几个强大的核心组成，而是由数千个较小的运算单元组成，能够同时执行许多运算。

[!NOTE]

“CPU 就像几个技艺精湛的大厨烹饪复杂的菜肴，而 GPU 就像成千上万个厨师同时准备简单的菜肴。”

2. gpu的架构

  1. 流式多处理器（SM）

    这些是主要的计算单元。每个 SM 包含许多小型算术逻辑单元（ALU）、寄存器和共享内存。

  2. CUDA cores

    负责执行加法和乘法等算术指令的基本计算单元。

  3. 内存层次结构

    • Global memory 全局内存（容量大但速度相对较慢）
    • Shared memory 共享内存（线程块内线程共享的快速内存）
    • Registers 寄存器（每个线程的超快存储）
    • Constant and texture memory 常量内存和纹理内存（专用缓存内存）

3. gpu图形起源

GPU 最初是为图形渲染而设计的，特别是用于将 3D 场景转换为 2D 图像等任务。

传统的渲染流程包括以下几个阶段：

  1. Vertex processing 顶点处理
  2. Geometry transformation 几何变换
  3. Rasterization 栅格化
  4. Pixel shading 像素着色

每个阶段都会对大量顶点或像素执行操作。由于这些操作大多是独立的，因此可以并行计算。

例如，计算每个像素的颜色涉及评估如下光照方程：

每个像素独立执行此计算，这使得 GPU 成为此类工作负载的理想处理器。

随着时间的推移，工程师们意识到同样的并行架构也可以加速非图形计算 。

4.gpu效率背后的数学原理

许多计算问题都可以用线性代数运算来表示，而 GPU 可以极其高效地执行这些运算。

一个基本的例子是矩阵乘法 。

5.线程层次：cuda运行模型

为了利用 GPU 的强大功能，开发人员使用 CUDA（统一计算设备架构） 等编程模型。

CUDA 将计算组织成一个层次结构：

Threads → Blocks → Grids 线程 → 块 → 网格

每个线程执行一小段计算。线程被分组为块 ，块构成网格 。

6.内存访问和性能

GPU 性能很大程度上取决于内存访问模式 。

全局内存访问速度相对较慢。因此，CUDA 程序会尽可能地利用共享内存和寄存器 。

一项关键的优化是内存合并 。

如果线程以连续模式访问内存：

A[i], A[i+1], A[i+2], …

GPU 可以将多个内存请求合并成一个事务。

然而，不规则的访问模式会导致许多单独的内存请求，从而降低性能。

实际上，经过良好优化的 GPU 程序与 CPU 实现相比，速度通常可提高 10 倍到 100 倍 。

7. 机器学习里的gpu

深度学习的兴起极大地提升了 GPU 的重要性。

训练神经网络需要反复进行矩阵运算。

例如，神经网络层计算：

y = W * x + b

where: 在哪里：

W = weight matrix W = 权重矩阵 x = input vector x = 输入向量 b = bias vector b = 偏置向量

在训练过程中，梯度下降需要计算导数：

dL/dW

并执行如下更新：

W = W − eta * dL/dW

这些操作涉及数十亿次浮点运算，而 GPU 由于其并行架构，能够极其高效地处理这些运算。

8.gpu在科学计算里的加速

CUDA 允许开发人员使用 C/C++ 的扩展来编写 GPU 加速程序。

现代 GPU 还支持以下高级功能：

  • Tensor cores for deep learning 用于深度学习的张量核心
  • Mixed-precision arithmetic 混合精度算术
  • Asynchronous execution 异步执行
  • Unified memory 统一内存

这些特性显著加快了矩阵乘法和卷积运算等任务的速度。

9.现代gpu创新

现代 GPU 的发展已经远远超越了图形硬件的范畴。

一些关键进展包括：

A. Tensor Cores

专为加速神经网络中使用的矩阵乘法运算而设计的专用单元。

D = A * B + C

合并成一条指令。

B. Ray Tracing Hardware

用于计算三维环境中光线相交的专用核心。

C. High Bandwidth Memory (HBM)

现代 GPU 采用堆叠式内存架构，可提供超过 1TB/s 的带宽。

这些改进使得 GPU 能够高效地处理海量数据集。

  • Climate simulations 气候模拟
  • Genomics 基因组学
  • Cryptography 密码学
  • Wireless communication research 无线通信研究
  • Computational physics 计算物理学

https://www.pi.website/research/real_time_chunking

PI05是同步执行的。推理-》执行-〉下一步；结果就是没有差异但是有停顿。

提出一种RTC的算法，能够实现无间断的实时执行，并且适用于任何基于扩散或流的 VLA（包括 π 0.5 ），无需任何训练时间的更改 。

一个优秀的实时算法应该生成一个与上次action chunk重叠动作一致的新数据块，同时保持模型对新信息的反应能力和做出智能决策的能力。

把实时分块问题建模为图像修复问题。假设我们的模型在接收到观测数据后，需要 3 个控制器时间步才能生成一个动作块。新动作块的前 3 个动作无法执行，因为在新动作块可用时，这 3 个时间步已经过去。因此，将这些动作“冻结”为已知会执行的前一个动作块的值是合理的；我们的目标是填充新动作块的剩余部分，就像修复图像中被移除的部分一样。

效果也非常好。

我们想要：把action expert 嫁接到vlm上，但是不需要丢失预先训练的网络规模只是，称之为“知识隔离vla”:利用 FAST 标记化的离散化动作来微调 VLM 主干网络，从而快速学习高质量的表示；同时，调整动作专家，使其能够生成连续动作，而无需将其梯度传播回 VLM 主干网络。训练完成后，动作专家可以通过流匹配生成流畅的连续动作，而离散化的动作则被丢弃。

1.阻止动作专家到vlm的梯度流

2.由于vlm不再接受action expert的梯度，表征不再适应机器人控制的要求，使得action expert更难生成正确的运动。用pi0-fast训练vlm骨干网络。
